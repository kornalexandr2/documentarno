# Руководство по запуску и разработке в Windows 11 (VS Code + Docker Desktop)

Это руководство описывает идеальный workflow для развертывания микросервисной архитектуры **Documentarno** на рабочей станции с Windows 11.

Благодаря использованию Windows Subsystem for Linux (WSL 2) и Docker Desktop, вам не нужно засорять саму операционную систему тяжелыми зависимостями (Python, CUDA, Node.js, C++ компиляторы). Вся разработка и выполнение происходят в нативных Linux-контейнерах с полным пробросом видеокарты.

---

## Шаг 1: Настройка Docker Desktop и WSL 2

Чтобы Docker мог использовать всю мощь вашего процессора и видеокарты:
1. Откройте **Docker Desktop**.
2. Перейдите в **Settings** (шестеренка сверху справа) -> **General**. Убедитесь, что включена опция **"Use the WSL 2 based engine"**.
3. Перейдите в **Settings -> Resources -> WSL Integration**. Убедитесь, что интеграция включена (если вы устанавливали дистрибутивы Linux из Microsoft Store, активируйте тумблеры для них, либо оставьте дефолтную интеграцию).
4. *(Для пользователей видеокарт NVIDIA)*: Убедитесь, что в самой Windows установлены свежие драйверы (Game Ready или Studio). Никакие дополнительные компоненты (CUDA Toolkit) в Windows ставить **не нужно** — Docker Desktop (через встроенный NVIDIA Container Toolkit в WSL 2) сам пробросит вашу видеокарту внутрь контейнеров Ollama и OCR.

---

## Шаг 2: Открытие проекта в VS Code

1. Откройте **Visual Studio Code**.
2. Нажмите `Файл` -> `Открыть папку...` (`File -> Open Folder...`) и выберите корневую папку проекта `Documentarno` (ту, где лежит `docker-compose.yml`).
3. Убедитесь, что у вас установлено официальное расширение **Dev Containers** (от Microsoft).
4. Откройте встроенный терминал: в верхнем меню выберите **Terminal -> New Terminal** (или нажмите `` Ctrl + ` ``). Можно использовать любой профиль (PowerShell, Command Prompt или Git Bash).

---

## Шаг 3: Первичная настройка окружения

Проекту необходимы переменные окружения для подключения к базам данных и настройки безопасности.
1. В панели файлов слева найдите файл `.env.example`.
2. Скопируйте его и назовите копию `.env` (сохраните в той же корневой папке).
3. При необходимости отредактируйте `.env` (например, измените `POSTGRES_PASSWORD` или `FIRST_SUPERUSER_PASSWORD`). Для локального запуска можно использовать значения по умолчанию.

---

## Шаг 4: Запуск проекта (Docker Compose)

Самый простой способ поднять всю инфраструктуру (Бэкенд, Фронтенд, PostgreSQL, Redis, Qdrant, Ollama, Celery) одной командой:

1. В терминале VS Code выполните:
   ```powershell
   docker compose up -d --build
   ```
2. **Процесс сборки:**
   - Docker скачает базовые образы баз данных.
   - Будут собраны кастомные образы для Backend (Python + зависимости) и Frontend (Node.js + сборка React).
   - *Примечание:* Первый запуск может занять **10-15 минут**, так как скачиваются объемные библиотеки для машинного обучения (PaddleOCR, PyTorch).
3. Когда в консоли появятся статусы `Started` для всех сервисов, подождите еще 1-2 минуты (бэкенд должен применить миграции базы данных Alembic и создать суперпользователя).
4. **Готово!** Откройте браузер и перейдите по адресу:
   👉 **`http://localhost:8080`**

*(Для остановки проекта используйте команду `docker compose down`).*

---

## Шаг 5: Разработка внутри Dev Containers (Опционально)

Если вы хотите писать код с полноценным автодополнением (IntelliSense) и линтингом *без установки библиотек в Windows*, используйте подключение к работающим контейнерам:

### Разработка Backend
1. Нажмите `F1` (или `Ctrl+Shift+P`) -> введите `Dev Containers: Attach to Running Container...`
2. В появившемся списке выберите контейнер `doc_backend`.
3. VS Code откроет **новое окно**, подключенное прямо внутрь Linux-контейнера бэкенда. Вы сможете редактировать Python-код, запускать тесты в терминале контейнера и использовать установленные там модули без конфликтов путей Windows.

### Разработка Frontend
Поскольку фронтенд в `docker-compose.yml` запускается через Vite с включенным Hot Module Replacement (HMR), вам не обязательно подключаться внутрь контейнера.
1. Редактируйте файлы в папке `frontend/src` прямо в вашей Windows (в основном окне VS Code).
2. При сохранении (`Ctrl+S`) изменения мгновенно отобразятся в браузере по адресу `http://localhost:8080` (папка `frontend` проброшена в контейнер через `volumes`).

## Возможные проблемы в Windows и их решение

### 1. Ошибка исполнения bash-скриптов ("CRLF vs LF")
Bash скрипты (`prestart.sh`, `install.sh`) чувствительны к окончаниям строк. В Windows по умолчанию используется формат `CRLF`. Если контейнер Бэкенда падает (например, с ошибкой `\r: command not found` или `exec user process caused: no such file or directory`):
- Откройте файл `backend/prestart.sh` в VS Code.
- В самом низу окна (в строке состояния) найдите надпись **CRLF**, кликните по ней и выберите **LF**.
- Пересоберите образ: `docker compose build backend`, затем перезапустите `docker compose up -d`.

### 2. Запуск без видеокарты (CPU-only)
Если команда `docker compose up` завершается с ошибкой, жалуясь на `capabilities: [gpu]` (потому что у вас нет видеокарты NVIDIA или драйверы установлены некорректно):
- Откройте `docker-compose.yml`.
- Найдите блоки `deploy:` (они присутствуют у сервисов `ollama` и `celery_worker`).
- Удалите или закомментируйте эти блоки полностью:
  ```yaml
  #  deploy:
  #    resources:
  #      reservations:
  #        devices:
  #          - driver: nvidia
  #            count: 1
  #            capabilities: [gpu]
  ```
- После этого система запустится в режиме использования только процессора (CPU).

### 3. Отсутствие ИИ-моделей
Поскольку Ollama запускается в чистом контейнере, изначально моделей там нет. Если вы попытаетесь задать вопрос в чате, получите ошибку. 
- Зайдите в веб-интерфейс Documentarno -> раздел **Model Hub**.
- Перейдите на вкладку "Скачать (Online)" и загрузите нужную модель (например, `llama3.1:8b` при наличии 8+ ГБ VRAM, или `qwen2.5:1.5b` для слабых систем).
